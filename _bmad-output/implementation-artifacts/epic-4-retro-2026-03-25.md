## Successes

1. **Pattern reuse from invites carried the epic** — Story 4.2 reused token hashing, public router mounting, BFF proxy, RLS migration, and findByTokenHash patterns from Epic 1's invite system. The backend was nearly mechanical to build. Investment in reusable patterns during Epic 1 paid compound returns.

2. **Code review caught 18 real issues** — Story 4.1 alone had an H1 (FAB rendering before AI completion), H2 (popover missing focus management), and H3 (wrong success feedback). None shipped. The adversarial review process is working.

3. **Testing maturity** — 82 new tests with zero flaky ones. Established patterns (`within()` scoping, `afterEach(cleanup)`, `matchMedia` mock, `navigator.clipboard` stub) are now team muscle memory. Nobody rediscovered how to test mobile vs desktop.

4. **Viral acquisition loop complete** — Share as PNG → shareable link → focused insight card → signup CTA. This is the portfolio showcase feature — a real product loop, not just plumbing.

5. **Discriminated union error handling** — Story 4.3's `fetchShare()` returns `{ status: 'ok' | 'not-found' | 'expired' | 'error' }` for clean branching. No nested try-catch.

6. **RSC + generateMetadata for OG tags** — Next.js 16 fetch deduplication means one HTTP call serves both the page render and metadata generation. Idiomatic and performant.

## Challenges

1. **Client-side complexity underestimated** — Story 4.1 was "just a PNG export" but the ShareMenu UI had real interaction requirements: focus management, click-outside-to-close, Escape key handling, per-action success feedback, FAB gating on AI completion. Seven code review findings from one "simple" component.

2. **Accessibility slipping on "simple" components** — The popover had `role="menu"` on a non-menu div, no focus trap, no click-outside dismiss. These are patterns we've handled before (TransparencyPanel in Story 3.6). False simplicity bred false confidence.

3. **Analytics schema can't track anonymous actions** — `analytics_events.userId` has a NOT NULL FK constraint. Anonymous share viewers can't be tracked. Worked around with atomic `viewCount` increment, but the schema design constrains future anonymous tracking.

4. **shadcn partially configured** — Epic 3's retro action item was to "configure shadcn properly." We installed Sheet but not Card or Button. Story 4.3 fell back to raw divs with Tailwind for the shared insight card. The action item was completed, but not comprehensively.

5. **SummaryText duplication** — The paragraph-splitting logic (5 lines) was knowingly duplicated from AiSummaryCard into SharedInsightCard. Pragmatic at 2 uses, but needs extraction at 3.

## Key Insights

1. **Pattern investment compounds** — The invite system from Epic 1 seemed like over-engineering for one feature. Two epics later, the same architecture carried shares with minimal new thinking. When building patterns, design for the second use case even if you only see the first.

2. **Client-side stories need full-stack rigor** — "No server code" doesn't mean simple. Interaction design, accessibility, state management, and cross-browser behavior all live on the client. Apply the same review discipline.

3. **Code review is the quality gate that works** — 18 findings across 3 stories, 100% addressed. The process catches cross-cutting bugs that unit tests miss. Mandatory review remains non-negotiable.

4. **RSC is the right default for read-only public pages** — Story 4.3's shared insight card is entirely server-rendered. No loading spinners, no client-side fetch, hits the 2-second NFR target. OG metadata works because generateMetadata runs server-side.
