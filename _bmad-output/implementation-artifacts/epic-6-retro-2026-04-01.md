## Successes

1. **Compound reuse effect** — Pattern investment from earlier epics accelerated every story. Story 6.1 established admin infrastructure (router mounting, roleGuard, BFF proxy, shadcn Table, JWT decode, types file, error boundary). Stories 6.2 and 6.3 followed the pattern with minimal new decisions. File counts shrank: 6.1 created 21 files, 6.2 created 5, 6.3 created 4.

2. **Zero blockers, zero debug sessions** — Cleanest epic in the project. Epic 2 had jsdom 28 issues, Epic 3 had curation pipeline complexity, Epic 5 had shared package rebuild friction. Epic 6 just worked.

3. **Code review producing reusable artifacts** — Story 6.1's review extracted `extractIsAdmin()` to `lib/auth-utils.ts` (reused in 6.2), moved duplicate types to `app/admin/types.ts` (extended in 6.2, 6.3), and added `error.tsx` error boundary (inherited by all admin pages). Review improved the codebase, not just the story.

4. **shadcn prep commit eliminated friction** — Epic 5 retro flagged shadcn setup as critical prep. It got done (Card, Button, Table pre-installed). All three Epic 6 stories used these components without setup delays.

5. **Shared package dev resolution fixed** — Epic 5's recurring pain point (manual `pnpm --filter shared build`) was resolved before Epic 6. Vitest aliases + tsconfig paths point to source. Three stories confirmed: no rebuild friction.

6. **"DO NOT Reinvent" tables as knowledge transfer** — Each story's dev notes included explicit tables of existing code to reuse, with file paths and patterns. This eliminated guesswork and kept implementation consistent.

## Challenges

1. **Architecture divergence on RLS** — The architecture doc specifies service-role DB connections (`dbAdmin`) for admin cross-org queries. All three stories used application-layer filtering (regular `db` instance + `roleGuard('admin')` + no `orgId` filter) instead. Pragmatically correct — RLS policies exist in migrations but middleware never calls `SET LOCAL app.current_org_id`. But the divergence wasn't formally caught until this retrospective.

2. **Pre-existing type-check errors still accumulating** — First noted in Epic 4, carried through Epic 5, still present in Epic 6. Story 6.2 notes "pre-existing type errors in `shared/types/transparency.ts` and `json` unknown in test files." Signal-to-noise erosion — developers stopped reading type-check output.

3. **Retro action item follow-through weak** — 0/6 items from Epic 5 retro fully completed. Some were scoped for Epic 7 stories (reasonable deferral). Others (Stripe type centralization, type-check cleanup) had no blockers but no home. The Epic 5 agreement "no more multi-retro carry-forwards" was itself carried forward.

4. **Story 6.3 status mismatch** — Sprint-status.yaml shows `done` but story file header says `Status: review`. Minor tracking inconsistency.

## Key Insights

1. **Code review catches implementation quality but misses architecture compliance** — Reviews verified Zod validation, type extraction, a11y, performance. Nobody checked "does this match the architecture doc's approach?" Adding an explicit checklist item closes the gap.

2. **Retro action items without a story home die** — Housekeeping tasks need to either become sub-tasks of the next story or mandatory prep items with a specific owner and deadline. "Nice-to-have" items get deprioritized indefinitely.

3. **Type-check cleanliness is a CI prerequisite** — Story 7.1 makes type-check a CI gate. Pre-existing failures must be cleaned first, or the gate can't be strict.

4. **Production ambition raises the bar** — Corey's intent to take this to production with real tenants elevated the RLS discussion from "nice-to-have defense-in-depth" to "required multi-tenancy infrastructure." This led to Story 7.6.

## Significant Discovery: Story 7.6 (RLS Middleware Activation)

Epic 6 revealed that all admin cross-org queries use application-layer filtering rather than the architecture doc's specified service-role DB connection. With production ambitions and real multi-tenant scale in mind, this gap must be closed.

**New Story 7.6: RLS Middleware Activation & Service-Role DB Connection**

Scope:
1. Auth middleware adds `SET LOCAL app.current_org_id` after JWT decode
2. `dbAdmin` connection pool with `BYPASSRLS` service-role user
3. Admin query files switch from `db` to `dbAdmin`
4. Docker Compose: non-superuser app role + superuser admin role
5. Integration tests proving RLS blocks cross-org reads on `db`, permits on `dbAdmin`
6. First-run Docker experience stays clean (`init.sql` creates both roles)

**Impact on Epic 7 sequencing:**

Original: 7.1 → 7.2 → 7.3 → 7.4 → 7.5
Revised: **7.6 → 7.5 (parallel) → 7.1 → 7.2 + 7.4 (parallel) → 7.3**

**Epic Update Required: YES** — Add Story 7.6, update sequencing.
