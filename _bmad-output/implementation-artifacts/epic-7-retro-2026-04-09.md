## Successes

1. **Portfolio readiness delivered** — README case study with architecture diagram, automated hero screenshots (light + dark), 5-stage CI pipeline, deterministic seed validation, dark mode. The "hiring manager runs `docker compose up` and sees a polished product" experience works. All portfolio artifacts are automated and reproducible — nothing manual goes stale.

2. **Story 7.6 (RLS activation) was the most architecturally significant story in the project** — 50+ files touched, every route handler wrapped in `withRlsContext`, dual DB connection pools, Docker dual-role initialization. Zero regressions. Three HIGH findings caught in code review (SSE transaction holding, missing checkout RLS, missing dataRows client param), all fixed. Production-grade multi-tenancy, not just portfolio-grade.

3. **Compound reuse effect peaked** — DO NOT Reinvent tables in every story file, established patterns from earlier epics, prior infrastructure (Playwright from 7.1, oklch tokens from earlier CSS work). Story 7.5 dark mode was mostly wiring because the CSS infrastructure was 95% done. Story 7.3 README screenshots were trivial because Playwright was already installed. Pattern investment from Epics 1-6 accelerated every Epic 7 story.

4. **Code review caught production-grade bugs** — H1: SSE streaming path held a database transaction open for 3-15 seconds (connection pool starvation under load). H2: Checkout route wasn't wrapped in withRlsContext (potential duplicate Stripe customers across orgs). These aren't style nits — code review is the most valuable quality gate in the process.

5. **Epic 6 retro prep work paid off** — Type-check restructuring (tsconfig.json/tsconfig.build.json split), Playwright + axe-core installation, Story 7.6 creation and sequencing — all executed before Epic 7 stories started. The "retro items become mandatory prep tasks" agreement worked.

6. **Test suite health** — 769+ tests across 75 files, all passing. Healthy testing pyramid: unit tests (Vitest) → integration tests (RLS enforcement, webhooks) → E2E (Playwright + axe-core) → CI (5-stage pipeline). New test categories added in Epic 7: RLS tenant isolation verification, seed validation snapshots, accessibility checks. Both API (445) and web (324) well-covered.

## Challenges

1. **Story file status tracking — 4-retro pattern** — Stories 7.4 and 7.5 say `Status: review` in file headers while sprint-status.yaml says `done`. This exact mismatch was flagged in Epic 4, carried through Epic 5, flagged as an action item in Epic 6 retro (not completed), and worsened in Epic 7. Root cause: dual-tracking (YAML + markdown headers) with no automation or ownership. Resolution agreed: fix all mismatches now and automate sync.

2. **ThemeToggle lint warning persists** — `react-hooks/set-state-in-effect` for `useEffect(() => setMounted(true), [])` pattern. Known since Epic 2 (React 19 `useSyncExternalStore` is the fix). Carried through every epic without resolution. Low impact but signals incomplete cleanup.

3. **Story 7.4 dev agent record is empty** — No completion notes, no change log, no file list. The story was implemented and reviewed, but the dev record section wasn't filled in. Minor documentation gap.

4. **CI pipeline unverified in real environment** — All 5 stages defined in `.github/workflows/ci.yml` but never triggered in GitHub Actions. First push may surface runner-specific issues (Docker Compose behavior, Playwright browser installation, env var handling, rateLimiter.test.ts flakiness). FR37 is theoretically verified, not practically verified.

5. **Uncommitted work** — Significant Epic 7 artifacts sitting in working directory: CI configs, E2E tests, scripts, snapshots, _explained.md docs. A hiring manager cloning the repo today wouldn't see any of it.

## Key Insights

1. **Process maturity is the project's real achievement** — From skipping validation in Epic 2 to zero-regression 50-file refactors in Epic 7. The 4-step story cycle (Create → Validate → Dev → Code Review), DO NOT Reinvent tables, mandatory code review, and prep tasks compounded over 7 epics and 35 stories. Each retro made the next epic smoother.

2. **High-priority retro items get done, low-priority ones don't — and that's honest prioritization, not failure** — Epic 6 action items #1 (architecture compliance) and #2 (prep task accountability) landed and directly enabled Epic 7 quality. Items #4 and #5 (status mismatches, Story 3.2) didn't land because they never ranked high enough. The fix is being ruthless: do it, promote it to a story, or explicitly drop it. No silent carry-forwards.

3. **Code review catches architecture-level bugs that testing can't surface** — Unit tests verify individual functions. Integration tests verify component interactions. But only a human reading the code in context caught "this SSE path holds a transaction for 15 seconds" and "this checkout route bypasses RLS." Code review is not a formality — it's the single most valuable quality gate.

4. **Portfolio readiness is infrastructure, not decoration** — Automated screenshots, deterministic seed validation, CI pipeline, dark mode with semantic tokens. Everything reproducible, nothing manual. The difference between "I built an app" and "I built a production-grade system" is in the infrastructure around the features.
