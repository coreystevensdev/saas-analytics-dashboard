## Successes

1. **Architectural pattern held for 4 consecutive stat types** — Every story in the cash-flow arc followed the same shape: discriminated union addition, pure function on aggregated inputs, formatStat + scoring + validator cases, prompt version bump with legal-posture framing. By the time 8.4 shipped, the pattern was mechanical — each layer's `case StatType.CashForecast` wrote itself. The discriminated union's exhaustive-switch check flagged every missing branch during development. Pattern investment from 8.1 and 8.2 made 8.3 and 8.4 low-risk despite their complexity.

2. **Privacy boundary tightened across the arc** — 8.1 established the pattern (pure function on `DataRow[]`). 8.3 tightened it further (public function takes scalars, private helper does row aggregation). 8.4 locked it in (`computeCashForecast` public signature takes zero `DataRow[]`; `monthlyNetsWindow` is the single seam). By the end of Epic 8, no public function in the curation pipeline accepts rows — the TypeScript compiler is the enforcement layer.

3. **Three-tier hallucination defense shipped and extended** — Tier 1 (currency/percent validator) shipped 2026-04-19 between 8.2 and 8.3. 8.3 extended it for break-even currency tokens. 8.4 extended it for forecast projected balances and deliberately excluded slope/intercept (regression coefficients the LLM should never quote). Tier 2 chart-reference validation shipped in 8.5. Tier 3 (input constraint — deterministic `crossesZeroAtMonth`, `confidence` tiers) is baked into every stat's details interface. LLM output fabrication risk is defended at every layer.

4. **"Validation pass" emerged as a new quality gate** — Story 8.4 ran a fresh-context validation against the story checklist BEFORE implementation. That pass caught 6 critical issues: wheel reinvention for `linearRegression` (already in project dependencies), wheel reinvention for `standardDeviation` (same), nonexistent query helper `getRecentRowsForCuration`, wrong analytics constants path, nonexistent SWR mutate pattern for cash-history, and a privacy boundary weakening in the `computeCashForecast` signature. All six were fixed in the spec BEFORE any code got written. This is a new companion to code-review in the 4-step story cycle — validation catches spec bugs, code review catches implementation bugs.

5. **Test suite pyramid extended cleanly** — 1,240 tests at epic close. Each new stat type came with unit tests (branches, suppression cases, boundary values), scoring tests (exact score assertion + monotonicity guard), assembly tests (format anchors), validator tests (allowed-set coverage + known tolerances), and end-to-end integration tests through the full pipeline. The end-to-end tests include privacy regression guards (no row labels in prompt) and ranking regression guards (runway > forecast > cashflow > break-even ordering). When the order flips, tests fail, forcing a rationale review.

6. **Prompt versioning discipline matured into a pattern** — Four version bumps (v1.3 → v1.4 → v1.5 → v1.6) across Epic 8. Every bump documented the cache-invalidation cost tradeoff explicitly in the story spec and the commit message. `DEFAULT_VERSION` is now a single-place update with automatic propagation via test assertions. The pattern of "extend v(N-1) verbatim, append new framing blocks, bump DEFAULT_VERSION" is mechanical enough that a new developer could do it without context.

7. **Point estimates over confidence bands was consistently right** — Runway is one number. Break-even is one number. Forecast is three numbers with a named crossing month. No fan charts, no ranges in prose unless confidence is 'low'. The decision to let confidence tiers soften framing rather than showing error bars is the most pro-owner call in the epic. Owners want a number to plan against; a fan chart is a statistics lesson they'll misread.

## Key Insights

1. **The discriminated union is doing architectural work, not just developer convenience** — Epic 8 added 4 new `StatType` branches to the `ComputedStat` union. Each addition forced `scoring.ts`, `assembly.ts`, `validator.ts`, and test files to add a `case` — the compiler was the linter. No silent fallthroughs, no default-case shortcuts masking missed cases. The cost of typing was low (~60 lines of type definitions per stat); the value was catching compile-time what would otherwise be runtime drift. This pattern will scale to 10+ stat types without breaking.

2. **"Validation pass" catches spec bugs; "code review" catches implementation bugs — both are mandatory now** — Story 8.4's validation found 6 critical spec bugs before a single line of code was written. Had those gone through to implementation, they'd have wasted hours of dev time and triggered code-review pushback. Validation isn't duplicate work — it's a different lens than code review. Validation reads the spec against the codebase; code review reads the code against the spec. Both surface different failures.

3. **Deterministic anchors beat LLM reasoning for hallucinable claims** — `crossesZeroAtMonth` computed in code; LLM paraphrases it. Confidence tiers enumerated; LLM selects tone from them. Every layer where the LLM could fabricate a number (month index, balance, percentage) is pinned to a pre-computed value. The validator doesn't catch month-index hallucinations — but the LLM can't make them up if the anchor is in its input. Tier 3 defense is the cheapest and most reliable hallucination mitigation: don't let the model reason about things it'll get wrong.

4. **Pattern replication compounds across stories — not just within epics** — 8.2 introduced `RunwayTrendChart`. 8.5 threaded it through `InsightChartThumbnail` + `InsightChartSheet`. 8.4 extended it with a `forecast` prop. Each story added capability without breaking the prior ones. `CashBalancePoint` became the shared shape that the forecast endpoint returns from the server (with aliased fields) so the client gets shape-compatibility for free. This compounding across stories — not just across epics — is why the final stories in Epic 8 shipped fast.

5. **Prompt-version cache-invalidation cost is a first-class concern** — Every bump costs real money (one full Anthropic generation per active org). Four bumps in Epic 8 means ~4× single-bump cost. No cost metrics deployed yet, so the actual spike is unobservable. The pattern of "deploy during low-traffic hours if cost-sensitive" is documented in every story's dev notes but there's no telemetry to verify it worked. This matters for the next prompt change — without metrics, we're flying blind on spend.
