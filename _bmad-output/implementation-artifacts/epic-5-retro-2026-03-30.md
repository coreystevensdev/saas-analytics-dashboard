## Successes

1. **Webhook handler pattern scaled cleanly** — Grew from 1 event type (checkout.session.completed) to 4 event types without restructuring the handler. The switch-case + dedicated handler function pattern from Story 5.1 carried the entire epic.

2. **Code review caught 18 real findings** — Consistent with Epic 4's rate. Caught schema partial index mismatches (H), dead code constants (M), Date precision bugs (M), responsive toast positioning (M), and missing logging (M). Not style nits — real issues.

3. **Idempotent webhooks by design** — `WHERE status != target` pattern is simple and bulletproof. No separate event ID tracking table needed. Replay is always a no-op.

4. **Story-to-story knowledge transfer** — Each story's dev notes included "Existing Code to Reuse — DO NOT reinvent" tables and "What This Story Does NOT Include" sections. Eliminated redundant work and scope creep.

5. **getActiveTier() evolved incrementally** — Story 5.1 created it (active only), Story 5.2 added canceled-within-period, Story 5.3 added test coverage for exclusion cases (expired, past_due). Each story built on the last without breaking anything.

6. **Pre-existing bug caught before production** — Story 5.2 discovered Story 5.1's missing unique constraint on `org_id`. The `upsertSubscription()` ON CONFLICT would have failed at runtime. Tests mocked the DB, so it passed in 5.1 but was caught during 5.2 implementation.

7. **Stripe test mode = production-identical code** — No conditional branches for test vs live. `sk_test_*` keys exercise the full Checkout flow, real webhook deliveries, real signature verification. Portfolio differentiator.

## Challenges

1. **Shared package rebuild friction** — Stories 5.2, 5.3, and 5.4 all hit the same issue: adding constants to `packages/shared/src/constants/index.ts` didn't propagate to `apps/api` tests until `pnpm --filter shared build` was run manually. The shared package compiles TS to `dist/`, and consumers import from `dist/`.

2. **Stripe SDK v20 type gaps** — `current_period_end` missing from `Stripe.Subscription` type (moved to `SubscriptionItem` in SDK v20, but webhooks still send it on top-level object). `subscription` missing from `Stripe.Invoice` type. Both stories created custom intersection types as workarounds.

3. **DB mock layer hid a real constraint violation** — Story 5.1's `upsertSubscription()` uses `onConflictDoUpdate({ target: subscriptions.orgId })` which requires a unique index. The schema only had a non-unique index. Tests mock the DB, so the ON CONFLICT never hit PostgreSQL. Caught in Story 5.2.

4. **Pre-existing type-check errors accumulating** — `TS18046` and `TS2532` in test files from Stories 5.2/5.3. Non-blocking but make `pnpm type-check` output noisy.

5. **Epic 4 action items carried forward again** — shadcn Card/Button setup (3rd retro) and Story 3.2 file status (4th retro) still not done.

## Key Insights

1. **Pattern investment keeps compounding** — Webhook handler, test factories, idempotent queries, BFF proxy routes — all established in early stories and reused without modification. The "DO NOT reinvent" tables in dev notes are the vehicle for this.

2. **Mock-heavy testing has a DB blind spot** — Fast and reliable (661 tests, zero flaky) but can't catch PostgreSQL-specific behavior like unique constraints or RLS policies. Scoped for Epic 7's CI pipeline.

3. **Shared package DX is a recurring tax** — Three stories hit the same rebuild friction. Fix is targeted: TypeScript `paths` in consumer tsconfigs pointing to `shared/src/` for dev, `dist/` for production builds.

4. **Subscription gate design proved correct** — Annotate, don't block. `subscriptionGate.ts` was unchanged across all 4 stories. `getActiveTier()` was the only query that evolved. The architecture decision held.

5. **Stripe test mode is a portfolio differentiator** — Production-identical code paths, real infrastructure, no mock payment service. Demonstrates production engineering judgment.
